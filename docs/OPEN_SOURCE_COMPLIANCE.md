# Open-Source Compliance Notes

This is a practical release checklist for VisionG. It is not legal advice.

## 1) Licensing model

- Project-owned VisionG code: `LGPL-3.0-or-later`
- In-tree third-party source: keeps original licenses
- External SDK binaries/headers: not relicensed by LGPL

## 2) Cleanup decisions

- Historical product-key enforcement flow was removed.
- `include/` keeps project-owned public headers only.
- Project-owned internal helpers live under `src/**/internal/` and `src/c_compat/include/`.
- Project-owned replacements now cover DMA allocation helpers and IVE memory/image helpers.
- SDK-derived sample source trees and vendor header snapshots are not kept in the repository.
- In-tree third-party source directories carry local license texts next to the retained code.
- SDK, RKNN, librga, and IVE headers are resolved from `--deps-root` or `--sdk-root`.

## 3) High-risk redistribution areas

Review separately before publishing anything beyond source:

- staged SDK/library bundles under `_stage/`
- staged target Python sysroot under `_stage/3rdparty/target_python/`
- local build outputs under `build/`
- locally produced release archives such as `visiong_cpp.zip` and `visiong_python.zip`

## 4) Header-license checkpoints

See `docs/INCLUDE_LICENSE_AUDIT.md`.
In short:

- many Rockchip MPI/RT headers are open or dual-licensed,
- some RKAIQ/RKNN headers still contain restrictive proprietary wording,
- treat staged RKNN/RKAIQ headers as distribution-sensitive unless you have explicit confirmation.

## 5) Public release checklist

Before publishing:

- keep `LICENSE`, `THIRD_PARTY_NOTICES.md`, and `licenses/` aligned with the released file set,
- prefer source archives generated by `scripts/create_source_release.sh`,
- keep `scripts/audit_dependency_policy.py` passing on the repository tree,
- keep `scripts/audit_binary_release_policy.py` passing on the repository tree,
- keep `scripts/audit_spdx_headers.py` passing on the repository tree,
- keep the generated archive passing `scripts/audit_source_release.py`,
- do not claim LGPL covers external SDK binaries/headers,
- do not publish staged dependency bundles or staged vendor libraries,
- only publish VisionG binary packages with a clear note that required Rockchip runtime libraries must be present on the target,
- preserve upstream attribution/copyright notices for in-tree third-party source.

## 6) Recommended posture

For clean public releases, publish:

- project source code,
- `visiong_cpp.zip`,
- `visiong_python.zip`,
- project-owned headers,
- documentation,
- dependency staging scripts,
- repository audit metadata.

Do not publish staged dependency bundles or staged vendor libraries.

See `docs/DEPENDENCY_REDISTRIBUTION_MATRIX.md` for the source/dependency class matrix and `docs/BINARY_REDISTRIBUTION_POLICY.md` for the binary release matrix used by the release process.
